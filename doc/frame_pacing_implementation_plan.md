# Frame Pacer — Implementation Plan

Work order for implementing the frame pacer designed in the planning-phase documents ([requirements](frame_pacing.md), [model](frame_pacing_control_model.md), [algorithm](frame_pacing_algorithm.md), [inputs](frame_pacing_inputs.md), [behavior](frame_pacing_behavior.md), [tiers](frame_pacing_capability_tiers.md)). This document adds sequencing, PR-sized steps, dependencies, and per-step acceptance — no new design. Gap references (G1–G8) are the numbered items of [frame_pacing_inputs.md](frame_pacing_inputs.md) §5.

Principles:

- **Every step lands independently** with its own verification; no step leaves the engine behaving worse than before it.
- **Observer before actuation** — the pacer computes and logs before it is allowed to enforce anything.
- **Parity is the gate** — the C++ pacer is not integrated until it reproduces the Python reference verdicts (C1–C9).
- **Instrumentation behind a facade** — the frame loop is under active refactoring (legacy single-cb path is being removed); record sites go through a thin recording interface so they can move without touching the pacer.

## Phase 4 — slop-servo fallback pacer (tier S) and method benchmark

Motivated by the review of the Games-by-Mason `FramePacer.zig` method (2026-07-22; method summary in [frame_pacing_capability_tiers.md](frame_pacing_capability_tiers.md) §2 "Planned tier S"): a backpressure-sensing sleep servo that needs no presentation-timing extensions, as the fallback when tier W/A requirements are not met (extension-less platforms, VRR displays, drivers with broken timing).

| Step | Work | Depends on | Acceptance |
|---|---|---|---|
| **P4.1 Slop-servo pacer (tier S)** | Sim-first: add a slop-servo pacer mode to the reference simulation (plant: plain-FIFO backpressure blocking as the sensed input; no feedback, no grid) with claims for its equilibrium (residual blocking ≈ headroom), overshoot back-off, and its documented failure modes (heavy-tailed load, `fifo_latest_ready`). Then a C++ `Slop_servo_pacer` beside `Frame_pacer`, fed from the existing involuntary-wait accounting (`cpu_service_time` subtraction already isolates slop), driving the same release-wait site at the editor tick. Tier resolution: W/A requirements unmet → S when plain FIFO + vsync available → else OFF. | P2.3 | Both suites green with the new claims; live on the development machine with tier W force-disabled: latency visibly reduced vs OFF, no death spiral, bounded behavior at heavy-tailed load |
| **P4.2 Pacer method owns present mode + image count** | The resolved pacer method decides the swapchain configuration instead of the current independent selector ranking: tier W/A → `fifo_latest_ready` (+ current image count, present timing enabled); tier S → **plain `VK_PRESENT_MODE_FIFO_KHR` + minimum image count** (the servo needs backpressure; latest-ready drops instead of blocking, slop reads ~0 and the servo winds to nothing). Wiring: tier resolution must happen before (or trigger) swapchain creation; expose the decision in the startup tier log and `get_frame_pacing_status`. | P4.1 | Swapchain mode/image count observably follow the resolved tier; switching the tier (config force) recreates the swapchain accordingly |
| **P4.3 Method quality benchmark** | A/B the two methods on the same scenes and induced loads (workload knob + light scene), same metrics from the existing records/MCP captures: input latency (release → visibility midpoint), visible-duration exactness, miss/drop rates, convergence after load steps, behavior at heavy-tailed and boundary-straddling loads. Document results in the tier doc §5 area (quality per tier), including where the simple method is good enough and where the full pacer's machinery pays. | P4.1, P4.2 | Comparison table with measured numbers in the tier doc; a defensible recommendation per platform/tier |

## Risks

| Risk | Mitigation |
|---|---|
| Local driver lacks `VK_EXT_present_timing` → Tier W unreachable on the development machine | P0.1 first — known within the first PR. All other phases proceed unchanged (observer mode and records work in OFF); actuation validation waits for a capable driver/machine. |
| Frame-loop refactoring moves instrumentation sites | Recording facade (P0.2); pacer never references engine internals. |
| Real input fidelity differs from the sim (service times, delivery latency) | Observer mode (P2.1) is a dedicated soak step before any actuation; C9 already bounds the delivery-latency case. |
| Windowed/DWM feedback semantics surprise | Validation signals (P0.3/P0.4) cross-check feedback before it is trusted; tier-doc §5 defines the degraded states. |
| Timer wake-up jitter larger than assumed | P0.5 measures before actuation; `guard` is a tunable. |
